== Building Archiva

Archiva itself builds with Bazel; see the top-level link:../BAZEL.md[BAZEL.md]
for the build, layout, and the +archiva_module()+ macro used by every Java
module in this tree.

From the repository root:

  # Build every module
  bazel build //...

  # Run every test (manual-tagged tests are excluded by default)
  bazel test  //...

  # Common deployables
  bazel build //archiva-cli:archiva-cli                            # CLI launcher
  bazel build //archiva-modules/archiva-web/archiva-webapp:archiva # archiva.war
  bazel build //archiva-jetty:archiva-jetty-dist                   # tar.gz distribution

== Publishing the pages for the archiva web content

This module and its children contain web content and project reports that can be
published to the archiva web site: https://archiva.apache.org

The web content parts of this module and submodules are published to the path

  /ref/${project.version}/

NOTE: The Bazel build does not (yet) cover site generation. The Maven Site /
SCM-Publish plugins are still used for this publishing-only workflow, driven
by the +deploySite.sh+ helper described below. Bazel has no direct equivalent
for the +site:stage+ / +scm-publish:publish-scm+ goals, so they remain the
authoritative path for refreshing the published documentation.

=== Use the script

There is a shell script +deploySite.sh+ which you can run to generate the site
check and publish to the remote repository. It works only on Linux, on other
platforms you have to invoke the underlying tools manually.

The script is interactive, it asks you to confirm the publish after generation
of the staging part.

.Execute

  ./deploySite.sh

All arguments are appended to the mvn calls performed by the script.

=== Some notes about the build process

A sparse checkout of the git repository will be created in

 .site-content

but only, if the directory +.site-content/.git+ does not exist.
